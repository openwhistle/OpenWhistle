Security Architecture

OpenWhistle is designed on a privacy-by-design principle (DSGVO Art. 25). Every layer of the stack is engineered to protect the whistleblower's identity — even from the system operator.

Four-Layer IP Anonymisation

A single misconfiguration should not be able to expose a whistleblower's IP address. OpenWhistle removes this risk through four independent, redundant layers of IP removal:

1

nginx — strip X-Forwarded-For

The nginx reverse proxy is configured to not forward any X-Forwarded-For or X-Real-IP headers to the upstream application. Even if a real IP header arrives from a CDN or upstream proxy, nginx removes it before passing the request to the app.

2

Application middleware — drop remote address

A custom ASGI/WSGI middleware intercepts every incoming request and replaces request.client.host (the TCP remote address) with a fixed placeholder value of 0.0.0.0 before the request reaches any route handler. No route handler ever sees a real IP address.

3

Database schema — no IP column

No table in the PostgreSQL schema has an IP address column. Not for reports, not for sessions, not for audit logs. There is no possibility of accidentally logging an IP because the column simply does not exist.

4

Redis sessions — no identifying metadata

Session records stored in Redis contain only the session ID (a random UUID), a user role, and an expiry timestamp. No IP address, no User-Agent string, no geolocation data. Rate-limiting buckets use hashed action tokens, not IPs.

Defence in depth

These four layers are independent. Even if nginx were misconfigured to forward the IP, the middleware would discard it. Even if the middleware had a bug, the schema has no column to store the value. Defence in depth means no single failure point can compromise anonymity.

Two-Factor Whistleblower Access

Whistleblowers do not create accounts. Instead, each submitted report generates two credentials that together act as a two-factor authentication for the message thread:

Credential Format Purpose Storage
Case Reference RPT-YYYYMMDD-NNNNN Identifies the report; readable but not guessable without brute-force over a very large space Stored in plain text (not secret on its own)
PIN UUID4 (36 chars, 122 bits of entropy) Acts as the authentication secret. Shown to the whistleblower once, never again Only a bcrypt hash is persisted in the database

Both values must be supplied together to access the /status page. The case reference alone is meaningless without the PIN. A UUID4 has 2122 possible values — brute-force is computationally infeasible, particularly combined with Redis-backed rate limiting on the /status endpoint.

Why not a 4-digit PIN?

A 4-digit PIN has only 10,000 possible values. An attacker who knows a case reference could enumerate all PINs in seconds. UUID4 eliminates this attack surface entirely while remaining copyable and storable by the whistleblower.

Admin Multi-Factor Authentication

All admin accounts require TOTP-based MFA. This is enforced at the framework level — there is no configuration option to disable it. The enforcement happens in two places:

  • Setup wizard: TOTP enrollment is a mandatory step before the first admin account becomes active. The account row is flagged as incomplete until the TOTP secret is confirmed with a valid 6-digit code.
  • Login flow: After password verification, every login session requires a TOTP code. Sessions are stored in Redis with a configurable expiry (default: 8 hours).

OIDC login is supported for organisations that use an identity provider (Keycloak, Azure AD, Okta, etc.). When OIDC is configured, the identity provider is responsible for MFA enforcement — OpenWhistle verifies that the OIDC provider's MFA claim is present in the token before granting access.

Compatible authenticators

Any RFC 6238-compliant TOTP application works: Aegis Authenticator (Android), Bitwarden Authenticator, Google Authenticator, Authy, or hardware tokens (YubiKey with TOTP support). Prefer open-source authenticators for the most trustworthy setup.

Redis Rate Limiting Without IP Tracking

Traditional rate limiting ties request counts to the client's IP address. OpenWhistle cannot do this without violating the IP anonymisation guarantee. Instead, rate limiting uses a combination of:

  • Action-scoped buckets: Each rate-limited endpoint has a sliding window counter in Redis keyed by a hash of the endpoint path plus a randomly rotated salt. This limits total submission velocity without identifying any individual requester.
  • Session-scoped limits for authenticated endpoints: Admin login attempts are rate-limited per username (not per IP). After 10 failed attempts, the account is temporarily locked for 15 minutes, enforced via Redis TTL.
  • CAPTCHA for public submission (optional): An optional hCaptcha integration can be enabled for the /submit endpoint. When enabled, hCaptcha's self-hosted mode is used to avoid external data transfer.

DSGVO-Compliant Resource Hosting

Loading resources from external CDNs (fonts, JavaScript libraries, analytics) causes the end user's IP address to be transmitted to a third party. A German court (LG München I, case 3 O 17493/20, January 2022) ruled that loading Google Fonts without explicit consent violates DSGVO Art. 6(1)(a).

OpenWhistle's approach:

Resource type Approach
Web fonts Fetched at build time from Bunny Fonts (EU-hosted, DSGVO-compliant CDN), then self-hosted within the Docker image. No runtime external font requests.
CSS & JavaScript All UI assets are bundled and served by nginx. No external library CDNs.
Analytics None. OpenWhistle contains zero tracking code.
Error monitoring Server-side only (structured logging to stdout), never client-side SDKs that phone home.

HTTP Security Headers

nginx is configured to send the following security headers with every response: 

Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
Content-Security-Policy:   default-src 'self'; script-src 'self'; style-src 'self'; font-src 'self'; img-src 'self' data:; connect-src 'self'; frame-ancestors 'none'
X-Frame-Options:           DENY
X-Content-Type-Options:    nosniff
Referrer-Policy:           no-referrer
Permissions-Policy:        camera=(), microphone=(), geolocation=(), payment=()
Cross-Origin-Opener-Policy: same-origin

The Content-Security-Policy blocks all external resources at the browser level, providing a client-side safety net that complements the server-side resource self-hosting.

HSTS preloading

In production, submit your domain to the HSTS preload list so browsers refuse any non-HTTPS connection before the first request is even made. This prevents SSL-stripping attacks.
