OpenWhistle Documentation

OpenWhistle is a self-hosted, stateless whistleblower reporting platform built for compliance with the German Hinweisgeberschutzgesetz (HinSchG) and EU Directive 2019/1937.

Architecture Overview

OpenWhistle follows a strict layered architecture. All components run as Docker containers managed by Docker Compose. The application is stateless — any state (sessions, rate-limit counters) is stored in Redis so that horizontal scaling or container restarts have no impact on users.

Key Design Decisions

Stateless application

The app container holds no in-memory state. Every restart is clean. Redis handles sessions and transient data; PostgreSQL handles persistent records. This allows zero-downtime deployments and horizontal scaling.

Four-layer IP anonymisation

IP addresses are neutralised at four independent points so that even a misconfiguration at one layer does not expose the whistleblower's identity. See the Security Architecture page for details.

UUID4 PIN system

After submitting a report, the whistleblower receives a random UUID4 string as their PIN. This is the only credential that links them to their case. The PIN is shown once and never stored in plain text — only a bcrypt hash is persisted. Combined with a short numeric case reference, the whistleblower can return to check for replies without creating any identifying account.

Mandatory TOTP MFA

Every admin account must enroll a TOTP authenticator (Google Authenticator, Aegis, etc.) before the account becomes usable. This cannot be disabled by configuration. OIDC-based login is supported for enterprise environments that provide MFA at the identity provider level.
