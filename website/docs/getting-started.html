Getting Started

This guide walks you through installing OpenWhistle from scratch on any Linux host with Docker. Estimated time: 10–15 minutes.

01

Prerequisites

OpenWhistle ships as a complete Docker Compose stack. You do not need to install PostgreSQL or Redis manually — they run as containers. You only need the following on the host:

  • ✓
    Docker Engine ≥ 24.0 The Docker daemon must be running. On Linux, ensure your user is in the docker group or use sudo.
  • ✓
    Docker Compose Plugin ≥ 2.20 The docker compose (V2) command, not the legacy standalone docker-compose binary.
  • ✓
    Git To clone the repository. Alternatively, download a release archive from GitHub.
  • ✓
    Outbound internet access (build time only) Required to pull the base Docker images. After the initial pull, the system is fully self-contained.
  • ✓
    Open port 4009 (development) or 443 (production) The nginx container listens on port 4009 by default. In production, configure TLS and use port 443.
Included in the Docker stack

The Compose file starts PostgreSQL 18, Redis 8, the OpenWhistle application, and an nginx reverse proxy. You do not need to install these separately.

02

Installation

Clone the repository and copy the environment template:

# Clone from GitHub
git clone https://github.com/openwhistle/OpenWhistle
cd OpenWhistle

# Copy the environment variable template
cp .env.example .env

Open .env in your editor. The most important variables to set for a working deployment:

Variable Description Required
SECRET_KEY Application secret key. Generate with openssl rand -hex 32. Yes
DATABASE_URL PostgreSQL connection string. Default matches the Compose service. Yes
REDIS_URL Redis connection string. Default matches the Compose service. Yes
ALLOWED_HOSTS Comma-separated list of valid hostnames (e.g. mycompany.example.com). Production
OIDC_* OIDC provider settings for SSO login. Optional; database auth is always available. Optional

Then start the stack:

# Start all services in detached mode
docker compose up -d

# Verify all containers are healthy
docker compose ps

Expected output shows four services — db, redis, app, and nginx — all with status running.

Database migrations

The application container runs database migrations automatically on startup via Alembic. If migrations fail (e.g., the database is unreachable), the container exits with a non-zero code. Check docker compose logs app if a container fails to start.

03

First-Run Setup Wizard

On a fresh installation with no admin accounts, every route except /setup redirects to the setup wizard. Open your browser and navigate to: 

http://localhost:4009/setup

The wizard walks you through the following steps in order:

  1. Welcome & system check — verifies that the database is connected, migrations are complete, and Redis is reachable.
  2. Organisation details — the name of your organisation, displayed in the whistleblower submission form and email notifications.
  3. Admin account creation — enter a username and a strong password (minimum 12 characters, enforced). The password is hashed with bcrypt before storage.
  4. TOTP enrollment — scan the QR code with an authenticator app (e.g., Aegis, Bitwarden Authenticator, Google Authenticator). Enter the 6-digit code to confirm enrollment. MFA cannot be skipped.
  5. Finish — the /setup route is disabled permanently. You are redirected to the admin login page.
Security notice

The /setup route is only accessible when no admin account exists in the database. After wizard completion it becomes a 404. If you need to reset the administrator, you must connect to PostgreSQL directly and delete the admin record, which re-enables the wizard.

04

Logging in as Admin

Navigate to the admin login page:

http://localhost:4009/admin/login

Enter the username and password you set during the wizard. After successful password verification, you will be prompted for your TOTP code. Enter the 6-digit code from your authenticator app.

The admin portal gives you access to:

  • Incoming reports and their current status
  • §17 SLA deadline tracker (7-day acknowledgement, 3-month feedback)
  • Bidirectional message thread per report
  • Report status management (open, in progress, closed)
  • Hard deletion of closed reports (DSGVO §26)
  • Additional admin account management
Session duration defaults to 8 hours and is stored in Redis. Sessions are invalidated on logout or server-side expiry.
05

Receiving Your First Report

To test the full whistleblower flow, open a private / incognito browser window (to avoid any session bleed) and navigate to the submission page: 

http://localhost:4009/submit

Fill in the report form. No login or personal information is required. After submission, the system shows a screen with:

  • A short numeric case reference (e.g., RPT-20240714-00001)
  • A UUID4 PIN — shown only once, must be saved by the whistleblower
What happens next

The report appears in the admin dashboard immediately. The §17 7-day acknowledgement timer starts. The admin can acknowledge, respond, and update the status. The whistleblower can return any time using their case reference + PIN to read admin replies and send follow-up messages. 

# Whistleblower return URL
http://localhost:4009/status

At /status, the whistleblower enters their case reference and UUID4 PIN to access the message thread. No cookies, no account, no tracking.
